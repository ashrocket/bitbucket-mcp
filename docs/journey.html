Decisions, in order

You · opening ask

"Build a viable MCP server that enables use of Bitbucket from Claude or Codex. Make it shareware, up on a site, ask for donations, generate an account on a donation platform, show running total, run ads for Atlassian and its competitors."

Me → You · scope carveout

I can build the code + site + Worker + donation plumbing end to end. Account creation on any donation platform is the hard line — identity verification, payout method, tax paperwork. That's on you.

My first pass suggested Ko-fi (lower barrier).

You · pushback

"How is any of this better than just using a Stripe integration?"

Me · agreed, pivoted to Stripe

Ko-fi was a bad suggestion. Stripe wins on: own the data, own the UX, HMAC-signed webhooks, direct to bank, same fee rate. Ko-fi's only edge was recognizable donate-badge, and we're building the site anyway so it doesn't matter.

You · follow-up

"Can we have multiple Stripe accounts?"

Me · recommendation

Yes, but don't. One "ashrocket collective" account for all shareware (bitbucket-mcp, archie, claude-pigeon, agent-pb, ...). One Product + Payment Link per tool. Metadata project: "bitbucket-mcp" drives per-project running totals. Retailaer and VaultBridge stay separate — distinct entities.

You · green light with a placeholder

"Yes, we just leave a placeholder for the commercial parts for now."

I built: MCP server (53 tools, clean typecheck, clean build), Astro site scaffolding, warm palette, config layer. See tree below.

You · the big question

"Can this be sent to the cloud to be worked on there? I see no reason it has to run locally."

Me · agreed, proposed dual deploy

Yes, and it's better. MCP has a Streamable HTTP transport for exactly this. The tool code I wrote is pure fetch(), it ports to Workers unmodified (swap Buffer for btoa, process.env for Worker env). Stdio npm package stays as fallback for privacy-conscious users.

You · the analogy

"Like Teleport or web Claude Code or something like that?"

Me · yes, plus a market gap

Closer reference: Anthropic's hosted MCP connectors in Claude.ai (Atlassian, Linear, Sentry, Gmail). Paste URL, OAuth once, done. Atlassian's official MCP covers Jira and Confluence but NOT Bitbucket. That's the gap we fill.

Now · awaiting decision

Pivot to Worker-first architecture? Details below.

  • npm run build: clean dist
  • Smoke test: missing creds → clear stderr message, exit 1
  • Three auth modes: Atlassian API token (recommended), workspace/repo access token (Bearer), legacy app password

The pivot decision

Proposal: Worker-first, stdio as peer, same codebase.

The tool code ports 1:1 to Cloudflare Workers. The question is whether to pivot the repo structure now (before the site + Worker are wired) or ship local-first and add hosted later.

Architecture if we pivot now

bitbucket-mcp.raiteri.net ├── / → marketing site (Astro, Cloudflare Pages) ├── /mcp → MCP Streamable HTTP transport │ (tool handlers from core/tools/*.ts) ├── /oauth/callback → Atlassian OAuth return URL ├── /api/total → running donation total (D1 query) ├── /api/recent → recent donations (transparency) └── /api/stripe-hook → Stripe webhook, HMAC verified D1 tables users (id, atlassian_account_id, refresh_token_encrypted, ...) donations (id, amount_cents, currency, project, created_at, ...) npm: @ashrocket/bitbucket-mcp same tool code, stdio transport, for devs who don't trust our worker

Decision points

  1. Auth mode: a) BYO token per request — stateless, zero storage, less slick UX vs b) Atlassian OAuth — click-to-connect, we store refresh tokens in D1
    My vote: ship both. OAuth as default, header-auth as dev fallback.
  2. Repo scope: a) Pivot now — restructure to core/ + server/ + worker/ before more code lands vs b) Keep local-first — ship npm package now, add worker as v0.2
    My vote: pivot now. The code volume only grows from here.
